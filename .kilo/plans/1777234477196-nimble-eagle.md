# glTF Support Assessment and Proposal

## Status
Implementation is not complete.

The repository already contains a partial CPU-side glTF importer and a generic runtime upload path, but it does not yet provide end-to-end glTF support in the running client.

## What Exists Today
- CPU-side import entry points exist in `include/stellar/import/gltf/Loader.hpp` and `src/import/gltf/Loader.cpp`.
- The cgltf-backed importer can parse scenes, nodes, meshes, materials, and images in `src/import/gltf/Importer.cpp`.
- Runtime scene upload and traversal exist in `include/stellar/graphics/RenderScene.hpp` and `src/graphics/RenderScene.cpp`.
- OpenGL resource upload exists in `src/graphics/opengl/OpenGLGraphicsDevice.cpp`.
- Basic importer regression coverage exists in `tests/import/gltf/ImporterRegression.cpp`.

## Blocking Gaps

### 1. glTF is not wired into the actual client path
- The client still creates a renderer through `create_renderer()` in `src/client/Application.cpp:28`.
- `src/graphics/RendererFactory.cpp:9` always returns `opengl::CubeRenderer`.
- `src/graphics/opengl/CubeRenderer.cpp` builds a hardcoded debug cube scene instead of loading `.gltf` or `.glb` assets.

Impact: imported glTF content is never used by the application.

### 2. Material and texture support is incomplete at render time
- `RenderScene` uploads images and materials, but the OpenGL backend only uses `base_color_factor` when drawing: `src/graphics/opengl/OpenGLGraphicsDevice.cpp:281-313`.
- No texture sampling is performed for base color, normals, or metallic-roughness.
- No alpha-mask, alpha-blend, or double-sided handling is applied.
- No lighting or PBR shading path exists.

Impact: even if a glTF scene were loaded, textured and PBR-authored assets would render incorrectly.

### 3. The current asset model is missing a real glTF texture/sampler layer
- `MaterialAsset` stores `base_color_texture_index`, `normal_texture_index`, and `metallic_roughness_texture_index` in `include/stellar/assets/MaterialAsset.hpp`.
- The importer fills those fields using `cgltf_texture_index(...)` in `src/import/gltf/Importer.cpp:296-323`.
- `SceneAsset` only stores `images`, `meshes`, `materials`, `nodes`, and `scenes` in `include/stellar/assets/SceneAsset.hpp`.
- There is no explicit imported texture object containing `source image`, `sampler`, and `texcoord set`.

Impact: glTF texture indices are not the same thing as image indices. This becomes incorrect as soon as a model has multiple textures, shared images, or non-default sampler settings.

### 4. Import coverage is only partial static-scene support
The importer currently reads:
- `POSITION`
- `NORMAL`
- `TEXCOORD_0`
- triangle indices
- a subset of PBR metallic-roughness material data
- external, buffer-view, and data-URI images

The importer does not currently read:
- tangents
- vertex colors
- additional UV sets
- joints and weights
- skins
- animations
- morph targets
- cameras or lights
- glTF extensions
- non-triangle primitives

Relevant code: `src/import/gltf/Importer.cpp:330-399` and `src/import/gltf/Importer.cpp:403-587`.

Impact: support is limited to a subset of static meshes and simple materials.

### 5. Runtime skinning and animation support are absent
- `stellar::scene::Node` has `skin_index`, but the importer never sets it and runtime never uses it: `include/stellar/scene/SceneGraph.hpp:34-41`.
- `SceneAsset` has no storage for skins, joints, inverse bind matrices, animations, samplers, or channels.
- `RenderScene` exposes mutable node transforms, but no glTF animation system exists.

Impact: animated or skinned glTF content cannot be represented or played.

### 6. Backend parity is not implemented
- The design claims runtime-selectable OpenGL/Vulkan support.
- The current graphics device factory is OpenGL-only: `src/graphics/GraphicsDeviceFactory.cpp:9`.
- No Vulkan graphics implementation files are present.

Impact: any glTF implementation done only for OpenGL still falls short of the repo's stated renderer parity goal.

### 7. Test coverage does not prove production-ready glTF support
Current tests only cover simple `.gltf` import with embedded/external images in `tests/import/gltf/ImporterRegression.cpp`.

Missing coverage includes:
- `.glb`
- textured rendering
- alpha modes
- texture/image index mapping correctness
- tangents and normals
- skinning and animation
- multi-scene assets
- backend parity

## Proposed Implementation Plan

### Phase 1. Finish a usable static glTF path
Goal: render real static glTF assets correctly in the client.

Tasks:
1. Replace or extend `CubeRenderer` so the client can load a `SceneAsset` from a configured `.gltf` or `.glb` file.
2. Introduce a texture asset layer in `SceneAsset` that separates:
   - imported images
   - imported textures
   - sampler state
   - texture coordinate selection
3. Update `MaterialAsset` to reference engine texture assets instead of raw cgltf texture indices.
4. Extend the importer to populate the new texture/sampler structures.
5. Update `RenderScene` and `GraphicsDevice` so material creation can bind the textures each material actually needs.
6. Replace the current flat color fragment path with at least a minimal textured PBR-compatible path:
   - base color texture/factor
   - normal map support if tangents are available
   - metallic-roughness inputs
   - alpha opaque/mask/blend handling
   - double-sided raster state handling
7. Add representative render/integration tests for a textured static glTF asset.

Expected outcome: static glTF scenes render in the client with materially correct appearance for the supported subset.

### Phase 2. Raise importer completeness for common real-world static assets
Goal: reduce failures on common DCC-exported models.

Tasks:
1. Import tangents when present.
2. Import vertex colors.
3. Import `TEXCOORD_1` if needed by material workflows.
4. Compute bounds during import instead of leaving defaults in `MeshPrimitive`.
5. Decide whether unsupported attributes should fail hard or degrade gracefully.
6. Add `.glb` regression coverage.

Expected outcome: robust support for common static asset exports from Blender and similar tools.

### Phase 3. Add animation and skinning support
Goal: support skeletal animated glTF scenes.

Tasks:
1. Extend scene asset types with skins, joints, inverse bind matrices, animation samplers, and animation channels.
2. Import `JOINTS_0` and `WEIGHTS_0` into a skinned vertex format.
3. Import glTF skins and attach `skin_index` to nodes.
4. Add runtime animation evaluation and skeleton pose updates.
5. Add GPU skinning or CPU skinning path.
6. Add animation regression tests and runtime validation scenes.

Expected outcome: skeletal animation is supported as a first-class glTF runtime feature.

### Phase 4. Reconcile with renderer parity goals
Goal: align glTF support with project architecture.

Tasks:
1. Either implement the same resource/material path in Vulkan.
2. Or explicitly narrow near-term scope so glTF support is considered OpenGL-only until Vulkan exists.

Expected outcome: project status matches documented promises.

## Recommended Order
1. Fix the asset model mismatch around textures and samplers.
2. Wire glTF loading into the client runtime.
3. Implement textured static rendering.
4. Expand importer coverage for common static cases.
5. Add animation/skinning.
6. Address Vulkan parity.

## Notes for Implementation
- The `STELLAR_ENABLE_GLTF` CMake option currently does not gate any behavior beyond a compile definition on `stellar_import_gltf`; no code path checks the macro. This is currently more of a seam marker than a feature toggle.
- The smallest high-value milestone is Phase 1 with static textured assets only.
- Full "glTF support" should be treated as phased work, not a single patch.

## Scope Decision
Selected direction: static support now, with animation/skinning explicitly deferred to a second milestone.

Implementation consequence:
- Phase 1 and Phase 2 define the initial milestone.
- Phase 3 remains planned follow-up work rather than a blocker for first delivery.
