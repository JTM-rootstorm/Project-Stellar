# Split VulkanGraphicsDevice Implementation

## Recommendation

`src/graphics/vulkan/VulkanGraphicsDevice.cpp` is a good candidate for a behavior-preserving split. The current 2320-line file already has clear concern boundaries: device setup, swapchain/render target management, pipeline creation, per-frame command recording, resource upload/destruction, and common helpers. Splitting along those boundaries should reduce review burden without changing the public `VulkanGraphicsDevice` API or runtime behavior.

## Target File Layout

Keep the public header at:

- `include/stellar/graphics/vulkan/VulkanGraphicsDevice.hpp`

Add private implementation files under `src/graphics/vulkan/`:

- `VulkanGraphicsDevice.cpp`
- `VulkanGraphicsDevicePrivate.hpp`
- `VulkanGraphicsDeviceCommon.cpp`
- `VulkanGraphicsDeviceDevice.cpp`
- `VulkanGraphicsDeviceSwapchain.cpp`
- `VulkanGraphicsDevicePipeline.cpp`
- `VulkanGraphicsDeviceFrame.cpp`
- `VulkanGraphicsDeviceResources.cpp`

## Proposed Ownership

`VulkanGraphicsDevice.cpp`

- Destructor
- `initialize`
- `is_initialized`
- `destroy_vulkan_objects`
- `allocate_handle`

Rationale: keep top-level lifecycle orchestration in the obvious entry file.

`VulkanGraphicsDevicePrivate.hpp`

- Include the public Vulkan graphics device header.
- Declare shared implementation-only helpers:
  - `stellar::platform::Error vulkan_error(const char* operation, VkResult result);`
  - `void log_vulkan_message(std::string_view message) noexcept;`
  - `std::uint32_t sanitize_dimension(int value, std::uint32_t fallback) noexcept;`
- Define `VulkanDrawPushConstants`, because both pipeline layout creation and draw command recording must agree on the exact push constant size/layout.

Rationale: avoid adding implementation helper declarations to the public header. Do not place embedded SPIR-V arrays here.

`VulkanGraphicsDeviceCommon.cpp`

- `vulkan_error`
- `log_vulkan_message`
- `sanitize_dimension`

Rationale: these are used by multiple implementation files and should have one definition to avoid duplicate code or ODR problems.

## Symbols To Handle Carefully

- `VulkanDrawPushConstants` must remain byte-compatible for both `vkCmdPushConstants` and `VkPushConstantRange::size`; centralize it in the private header.
- `vulkan_error` is currently used throughout the file; make it a single shared definition, not copied anonymous-namespace functions in each `.cpp`.
- Keep `kSolidVertexShaderSpv` and `kSolidFragmentShaderSpv` out of headers to avoid duplicate storage and compile overhead.
- Preserve the current destruction order in `destroy_vulkan_objects`:
  - wait device idle
  - destroy meshes/textures/materials
  - destroy swapchain resources
  - destroy pipeline layout
  - destroy sync objects
  - destroy command resources
  - destroy device/surface/instance
- Preserve `destroy_sync_objects` before `destroy_command_resources`; sync teardown iterates `frames_`, and command teardown clears it.
- `valid_image_format` appears unused in the current source. Do not mix dead-code cleanup into the split unless the implementation pass explicitly decides to remove it in a separate cleanup step.

## Build System Changes

Update `CMakeLists.txt` in the `stellar_graphics` source list to include the new `.cpp` files:

- `src/graphics/vulkan/VulkanGraphicsDeviceCommon.cpp`
- `src/graphics/vulkan/VulkanGraphicsDeviceDevice.cpp`
- `src/graphics/vulkan/VulkanGraphicsDeviceSwapchain.cpp`
- `src/graphics/vulkan/VulkanGraphicsDevicePipeline.cpp`
- `src/graphics/vulkan/VulkanGraphicsDeviceFrame.cpp`
- `src/graphics/vulkan/VulkanGraphicsDeviceResources.cpp`

Keep `src/graphics/vulkan/VulkanGraphicsDevice.cpp` in the target.

## Implementation Steps

1. Add `VulkanGraphicsDevicePrivate.hpp` with shared declarations and `VulkanDrawPushConstants`.
2. Move common helper definitions to `VulkanGraphicsDeviceCommon.cpp`.
3. Move device/instance/sync functions to `VulkanGraphicsDeviceDevice.cpp`.
4. Move swapchain/render-pass/depth/framebuffer functions to `VulkanGraphicsDeviceSwapchain.cpp`.
5. Move pipeline layout/pipeline creation plus embedded SPIR-V to `VulkanGraphicsDevicePipeline.cpp`.
6. Move frame begin/draw/end functions to `VulkanGraphicsDeviceFrame.cpp`.
7. Move mesh/texture/material upload and resource helper functions to `VulkanGraphicsDeviceResources.cpp`.
8. Leave all member declarations in `VulkanGraphicsDevice.hpp` unchanged unless includes need a small adjustment after compilation.
9. Update `CMakeLists.txt` to compile the new translation units.
10. Build and fix include/linkage issues only; avoid behavioral refactors in the split change.

## Verification

Run the normal build and tests after the split:

```bash
cmake --build build -j$(nproc)
ctest --test-dir build --output-on-failure
```

If the build directory is not configured or Vulkan context tests are not enabled, configure a separate validation build or enable `STELLAR_ENABLE_VULKAN_CONTEXT_TESTS` and run:

```bash
ctest --test-dir build -R '^vulkan_context_smoke$' --output-on-failure
```

## Scope Control

This should be a pure file split with no intended behavior changes. Defer these follow-ups unless requested separately:

- introducing RAII wrappers for Vulkan handles
- changing memory allocation strategy
- changing shader module handling
- removing unused helpers
- splitting the public header/private member declarations into a full PIMPL design

A PIMPL refactor could make the public header smaller, but it is a much larger change and is not necessary to solve the immediate maintainability problem.
